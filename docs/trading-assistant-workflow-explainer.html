§ 02Monorepo architecture

The assistant workflow still has three central packages: packages/trading_assistant for the control plane, packages/trading_assistant_data for data contracts and source adapters, and packages/trading_assistant_backtest for replay and optimizer evidence. monorepo bot surfaces

Production source of truth is now ported under bots/. k_stock_trader runs KALCB/OLR; ibkr_trading has stock, momentum, and swing portfolios; crypto_trader owns crypto. Decides orders, emits telemetry, owns strategy code. Never receives commands from the assistant.

trading_assistant

Control plane. Curates evidence, schedules, freezes manifests, runs deterministic gates, routes approvals, owns the ledgers. No replay engines, no orders.

trading_assistant_backtest

Replay + optimization lab. Reads manifests, validates frozen baselines and evidence bundles, emits retained phase/repair attempts, proves parity for crypto, k_stock, and trading stock/momentum/swing bridge surfaces, and keeps approval blocked until contracts are approval_ready. No approvals, no orders.

Key principle (from the implementation plans): the backtest repo may discover candidates; only trading_assistant may trust, gate, review, and route them. Replay against live strategy code is the only path that can produce an approval-ready structural candidate.

§ 04Daily workflow — sensor + diagnostic layer

Daily runs are the system's eyes and ears. They do not propose authoritative strategy changes; their job is situational awareness, anomaly detection, process-quality checks, and lightweight hypotheses for the weekly synthesizer.

Inputs

  • raw bot events from EventQueue (TradeEvent, MissedOpp, DailySnapshot, ErrorEvent)
  • per-bot timezone from BOT_TIMEZONES — daily cron fires at market close + delay
  • last 30d findings (corrections, patterns) injected into prompt

Outputs

  • runs/<task_id>/daily_report.md + checklist
  • parsed StrategySuggestions written to suggestions.jsonl
  • process_quality, drift, hourly, fill_quality records
  • telegram/discord/email digest gated by report_checklist

§ 05Weekly workflow — synthesis & bounded steering

Weekly runs on Sunday cron. They aggregate seven days of evidence, run cross-bot pattern detection, recalibrate scorecards, and emit a single bounded artifact for the monthly run: monthly_search_brief.json. Weekly cannot trigger OOS repair, change the monthly sequence, or approve a candidate.

Guardrail — the brief is generated in report-only mode before runners consume it. One-week noisy signals are confidence-capped. Every weekly hint that influences candidate ordering must be attributed back to a monthly outcome so the brief itself can be scored.

Weekly portfolio focus rotation

The weekly learning loop should rotate the evidence focus by strategy family, while monthly phased-auto and OOS repair remain the authority for material optimization. Each weekly slice curates telemetry, missed opportunities, regressions, candidate neighborhoods, and negative priors for the next monthly run; it does not approve or deploy changes by itself.

§ 06Monthly authoritative loop

This is where material strategy improvement is intended to happen. The orchestrator freezes a MonthlyRunManifest → invokes the backtest workspace as a subprocess → validates emitted artifacts → routes a model review and a human approval. MonthlyArtifactContract is the control-plane owner for artifact paths, scope, containment, approval-gate evidence, and verifier inputs; the backtest side keeps the execution internals under trading_assistant_backtest.monthly_execution; the data side keeps canonical slice writes, coverage views, and authority under trading_assistant_data.slices. Today this path is a fail-closed shadow-ready spine: it can consume committed authoritative portfolio bundles, validate shadow-mature plugin contracts across the active bridge surfaces, retain phase/repair artifacts, pass the five validation tests, pass P6/P7 optimizer evidence checks, and emit deterministic no-adoption evidence. Weekly focus rotates across k_stock/trading-stock, trading-momentum, trading-swing, and crypto, but phased-auto optimization and OOS repair remain monthly, manifest-frozen, and approval-gated. It still cannot adopt an approval-ready candidate until at least one bridge is promoted to approval_ready with runtime/VPS metadata, broader production fixtures, scheduled shadow evidence, and production strategy parity.

Runner CLI: how invocation is actually wired today

The control plane's orchestrator/backtest_invocation.py builds the subprocess command from MonthlyRunManifest.backtest_command. In the current package layout, BACKTEST_REPO_PATH resolves to packages/trading_assistant_backtest; package-local examples may use ../trading_assistant_backtest from packages/trading_assistant. If monthly_backtest_command (env MONTHLY_BACKTEST_COMMAND) is left empty, the fallback remains python -m backtests.shared.monthly_repair --manifest <path> — the documented compatibility wrapper. The native python -m trading_assistant_backtest.monthly --manifest <path> form is also supported and is the preferred explicit command for new invocations. The test fixture in tests/test_app_wiring.py uses yet another command (python -m skills.backtest_runner_client) for in-process validation, which is not a production default.

Data handoff: what is wired today

The current checkout no longer relies on a committed May crypto bundle under packages/trading_assistant_data/data/bundles/monthly/.... Historical data and replay evidence are ported into monorepo-owned bot data directories, backtests/data_portability_manifest.json, backtests/baselines/baseline_index.json, and normalized validation artifacts. MonthlyValidationRequest and MonthlyRunManifest still accept direct data_bundle_manifest_path and optional data_bundle_checksum for compatibility, and validators prefer those fields when a real bundle is supplied; the current final-refactor acceptance path verifies portability, latest-round no-drift, optimizer equivalence, replay evidence, and parity from the frozen baseline/data-portability spine.

Scoring contract: what is binding today

The monthly runner resolves a strategy or portfolio-specific immutable score profile from trading_assistant_backtest.scoring.immutable. Candidate ranking, fold summaries, selection-OOS checks, repair comparisons, and objective_breakdown.json use that profile's normalized components and hard rejects. The emitted score payload carries effective_objective_version=immutable_score_profiles_v1, objective_profile_id, score_component_cap, renormalized_components, reject_reasons, and profile metadata; the control plane lifts those fields into selected candidates, gate reports, and approval packets. objective_weights_v1 remains stable only as a legacy helper composite for learning snapshots and local triage.

Sequencing rules (target state)

  1. Run diagnostics on round_N first. Never search before the incumbent's failure modes are understood.
  2. Two-fold purged folds on in-sample. Latest completed month stays outside fold scoring — it is selection-OOS, not a clean deployed verdict.
  3. OOS repair is conditional. Trigger only when selection-OOS materially trails IS/fold expectation. Weekly hints never trigger repair.
  4. Cumulative ablation. Repair must examine accepted mutations from all prior rounds, not just round_N.
  5. Confirmatory rerank centers on the repair winner (or phased-auto winner if no repair). All candidates compete under the resolved immutable score profile and preserve effective_objective_version plus objective_profile_id.
  6. Exactly one backtest candidate becomes round_N+1, or a deterministic no-adoption reason is emitted.
  7. The next completed month after deployment is the primary clean verdict. 3-month or min-trade follow-up confirms persistence.

§ 08Learning loop & feedback

The system has two stacked learning layers. The trading authority layer uses monthly full-fidelity evidence to change strategy. The harness meta-learning layer measures the analysis itself — prompts, retrieval, validators, parsers, generated memory, provider routing — against an executable benchmark, and accepts changes only when the benchmark gains hold.

The loop-engineering port adds a compact bridge between those layers: performance_learning_ledger.jsonl projects source-backed strategy and portfolio lessons from proposals, change records, monthly verdicts, portfolio outcomes, artifact indexes, search briefs, and priors. It feeds bounded EvidenceMemory/ContextBuilder summaries and future priors/search allocation, while preserving cadence authority so daily/weekly/harness evidence cannot masquerade as monthly approval evidence.

Why two layers and not one: a single layer would let prompt churn masquerade as performance gains. By keeping trading-authority changes gated on full-fidelity replay AND harness changes gated on an executable benchmark, the system can improve how it thinks without weakening the gates on what it changes in production.

§ 09Five-layer memory model

Memory is split by authority. Higher layers are stable and edited only by humans; lower layers are generated, scored, and decayed automatically. A change in the lower layers cannot escalate to a higher one without passing the corresponding gate.

The loop-engineering port adds compact projections over those layers: loop_run_ledger.jsonl is an operating feed from scheduled-run state, run metadata, proposal links, and monthly artifacts; work_log.md is generated from that projection for humans and future agents; performance_learning_ledger.jsonl is the bounded decision/outcome memory used for strategy and portfolio lessons. None of these files owns lifecycle, approval, or source-ledger truth.

§ 10Contracts & manifests — the seams between packages

The three package workspaces are coupled through frozen JSON manifests, not in-process imports. The control plane owns the canonical Pydantic schemas and validators; the data and backtest workspaces currently carry compatible local/vendored models and must stay contract-conformant through tests. The local shape is packages/trading_assistant, packages/trading_assistant_data, and packages/trading_assistant_backtest, each with package-local pyproject.toml, src/, and tests/; legacy pre-package checkout paths are historical/migration-only and are normalized by loaders where needed.

Loop-engineering adds a second contract plane inside the control package: memory/loops/*.md defines each recurring loop's cadence, authority, inputs, outputs, required checks, and stopping criteria; memory/artifacts/registry.yaml classifies which artifacts are binding, advisory, diagnostics-only, generated, human-owned, or eligible for approval gates.

The validation chain (what the control plane checks)

  1. tools/check_loop_contracts.py — validates scheduler-to-loop-contract alignment, authority boundaries, artifact registry coverage, delivered ledger/work-log evidence, diagnostics-only model-review rejection, outcome authority, and retired WFO wording.
  2. tools/check_performance_learning_ledger.py — validates source-backed performance-learning projections, strict source JSONL, stable event identity, stage separation, strategy/portfolio coverage, and relevance-scoped prompt summaries.
  3. skills/backtest_runner_client.py — captures stdout/stderr/exit_status, requires artifact_index.json, enforces path containment, freshness vs run_manifest.json, JSON/JSONL parsability, manifest id echo, data bundle checksum echo.
  4. skills/monthly_optimizer_runner.py — Phase-4 optimizer sequence: core artifacts, two-fold window alignment, search-brief consumption, candidate attempts, confirmatory rerank, rounds manifest, selected candidate lineage, runner contract version, structural parity, plugin maturity.
  5. skills/artifact_authority_registry.py — keeps advisory, diagnostics-only, generated, and human-owned artifacts out of actionable approval support while allowing deterministic binding and approval-gate evidence.
  6. skills/monthly_evidence_verifier.py — reads the packet, model-review validation, gate reports, deployment blockers, and evidence paths before approval routing; fail or needs_human_review suppresses automated approval readiness.
  7. contracts.validate_monthly_runner + the conformance fixture — same checks must pass against the real backtest command before any approval routing.
Current contract note: MonthlyRunManifest and MonthlyValidationRequest both carry data_bundle_manifest_path and data_bundle_checksum. The validators prefer the direct bundle fields when present, while market_data_manifest_path remains available for the legacy single-slice coverage compatibility path.
